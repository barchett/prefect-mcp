## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP caller -> Prefect MCP server (stdio) | Tool args arrive as MCP JSON-RPC payloads validated by Zod before any SDK call |
| Prefect MCP server -> OpenCode HTTP API (localhost) | Bodies forwarded to OpenCode; OpenCode enforces auth, tool gating, and message handling |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-10-01 | Tampering | Zod schema for `tools` field | mitigate | `z.record(z.string(), z.boolean())` rejects non-boolean values, malformed records, and arrays at the schema layer before any SDK call |
| T-10-02 | Tampering | Zod schema for `files` field | mitigate | `z.array(z.object({ type: z.literal('file'), mime: z.string(), filename: z.string().optional(), url: z.string() }))` rejects unknown shapes; non-file types are blocked by the literal('file') discriminator |
| T-10-03 | Information Disclosure | `files[].url` accepting arbitrary URIs (e.g., `file:///etc/passwd`) | accept | OpenCode runs locally on the same host as the caller; the MCP client (Claude Code) already has filesystem access. The URL is read by OpenCode's own file-loading code, which applies its own sandbox; Prefect adds no new disclosure surface |
| T-10-04 | Elevation of Privilege | `tools` field could enable a tool the user never normally grants | accept | The `tools` field can only enable tools that the OpenCode instance already has installed and configured. It cannot grant access to non-existent tools.
