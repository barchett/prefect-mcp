POST /session/{id}/message — opencode_run:
```typescript
// Request body:
{
  parts: Array<{ type: 'text', text: string }>;  // for our tool we always send a single text part
  messageID?: string;
  model?: { providerID: string; modelID: string };
  agent?: string;
  noReply?: boolean;
  system?: string;
  tools?: { [key: string]: boolean };
}
// Response: { info: AssistantMessage, parts: Part[] }
// CRITICAL: Long-lived HTTP — blocks until full agent loop completes (potentially minutes).
// CRITICAL: Do NOT pass `signal` or AbortController. RESEARCH.md Pitfall 2.
```

GET /session/{id}/diff — opencode_get_diff:
```typescript
// Query params: messageID?: string  (optional)
// Response: FileDiff[] where FileDiff = { file, before, after, additions, deletions }
```

POST /session/{id}/permissions/{permissionID} — opencode_approve_permission:
```typescript
// Request body: { response: "once" | "always" | "reject" }
// Response: boolean
// CRITICAL: REQUIREMENTS.md says allow/deny/allow_always — that's WRONG.
// The API enum is "once" | "always" | "reject". RESEARCH.md Pitfall 3, verified from SDK types.
```

POST /session/{id}/fork — opencode_fork:
```typescript
// Request body: { messageID?: string }  (optional — fork from current tip if omitted)
// Response: Session
```

POST /session/{id}/revert — opencode_revert:
```typescript
// Request body: { messageID: string, partID?: string }   (messageID is REQUIRED)
// Response: boolean
```

SDK call shape (from existing src/index.ts written in Plan 02):
```typescript
const { data, error } = await client.session.<methodName>({

});
if (error) throw new Error(JSON.stringify(error));
return { content: [{ type: 'text', text: JSON.stringify(data) }] };
```

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Claude Code -> MCP server | JSON-RPC over stdio; same-machine trust domain |
| MCP server -> OpenCode HTTP | Local HTTP to OPENCODE_URL (default 127.0.0.1:4096); no auth per RESEARCH.md security domain |
| Claude Code -> opencode_run prompt | The `prompt` string is forwarded verbatim to OpenCode as the user message — OpenCode is the LLM/agent runtime that interprets it |
| Claude Code -> opencode_approve_permission | Tool can grant `always` permissions in OpenCode — broader trust effect than `once` |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-03-01 | T (Tampering) | opencode_approve_permission enum | mitigate | Zod `z.enum(['once', 'always', 'reject'])` rejects any other value at the MCP boundary; can't accidentally send `"allow_always"` and trigger an OpenCode 400. Verified by acceptance criteria grep for the exact enum literal. |
| T-01-03-02 | T (Tampering) | opencode_revert messageID required | mitigate | Zod schema marks messageID as required (no `.optional()`); calls without it fail at the MCP boundary with a clear schema error rather than reaching OpenCode and 4xx-ing. Verified by acceptance criteria. |
| T-01-03-03 | I (Information Disclosure) | Diffs returned to Claude Code | accept | `opencode_get_diff` returns full file before/after content. This is by design — Claude Code needs the diff to review. Same trust domain. RESEARCH.md security domain confirms no ASVS controls. |
| T-01-03-04 | D (Denial of Service) | opencode_run blocking forever | accept | The blocking behavior is correct (RESEARCH.md Pitfall 2). If OpenCode hangs, the tool call hangs — Claude Code's existing tool-timeout handling at the orchestration layer is the user's recourse, not something this MCP server should override. |
| T-01-03-05 | E (Elevation of Privilege) | opencode_approve_permission with "always" | accept | The user has configured OpenCode to auto-approve operations (PROJECT.md Key Decisions); `opencode_approve_permission` is a pass-through tool. Granting "always" has no broader effect than the existing config. Personal-use trust model. |
| T-01-03-06 | S (Spoofing) | sessionId passed by Claude Code | accept | OpenCode session IDs are ULIDs; an invalid one returns a 4xx, surfaced as `isError: true`. Claude Code is in the same trust domain — no adversarial spoofing concern. |
</threat_model>
