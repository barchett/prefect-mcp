Two independent additions that both touch `src/index.ts` and `package.json`. Bundled together because (a) they are small, (b) they share files, and (c) keeping them in one wave-3 plan avoids three plans contending for the same files in wave 2.

1. **SURF-01**: Add `patch: string` to each element of the `opencode_get_diff` response, computed via the `diff` npm package's `createPatch(filename, before, after)`. The OpenCode API does NOT return a patch string (verified — `FileDiff = { file, before, after, additions, deletions }`), so we compute it client-side. Per CONTEXT D-04, hand-rolling unified diff is exactly what trips on edge cases (no-newline-at-eof, binary, empty files); the `diff` library is pure JS, no native bindings.

2. **CMD-01**: Register a new `opencode_session_command` tool that calls `POST /session/:id/command` to run slash commands inside a session (e.g. `/compact`, `/clear`). Per D-19, the model field here is a plain `string`, NOT `{ providerID, modelID }` — that is a deliberate API difference from `opencode_run` and must be preserved.

Purpose: Callers can navigate diffs without raw JSON parsing (the `patch` field is directly usable), and Claude Code can run slash commands inside sessions to compact context, clear history, etc.

Output: Updated `package.json` (with `diff` + `@types/diff`) and updated `src/index.ts` (with modified `opencode_get_diff` and new `opencode_session_command`).

```typescript
// CMD-01: Run a slash command inside an OpenCode session (e.g. /compact, /clear).
// Calls POST /session/:id/command. Same response shape as opencode_run:
// { info: AssistantMessage, parts: Part[] }. Note that `model` here is a plain
// string (e.g. "anthropic/claude-3-5-sonnet"), NOT a { providerID, modelID }
// object — this is deliberate; the OpenCode command endpoint accepts a single
// model string, unlike the prompt endpoint.
server.registerTool(
  'opencode_session_command',
  {
    description:
      'Run a slash command inside an OpenCode session (e.g. compact, clear). Returns { info: AssistantMessage, parts: Part[] } as JSON. Use this for session-level operations that have no equivalent SDK method.',
    inputSchema: z.object({
      sessionId: z.string().describe('Session ID'),
      command: z.string().describe('The slash command name without the leading slash (e.g. "compact")'),
      arguments: z.string().describe('Arguments string to pass to the command (use empty string if none)'),
      messageID: z.string().optional().describe('Optional message ID for context'),
      agent: z.string().optional().describe('Optional agent override'),
      model: z
        .string()
        .optional()
        .describe('Optional model override as a plain string (NOT { providerID, modelID } — this endpoint takes a single string).'),
    }),
  },
  async ({ sessionId, command, arguments: args, messageID, agent, model }) => {
    try {
      const { data, error } = await client.session.command({
        path: { id: sessionId },
        body: {
          command,
          arguments: args,
          ...(messageID ? { messageID } : {}),
          ...(agent ? { agent } : {}),
          ...(model ? { model } : {}),
        },
      });
      if (error) throw new Error(JSON.stringify(error));
      return { content: [{ type: 'text', text: JSON.stringify(data) }] };
    } catch (err) {
      return { content: [{ type: 'text', text: String(err) }], isError: true };
    }
  }
);
```

Concrete details (do NOT deviate):
- The handler destructures `arguments: args` because `arguments` is a reserved word in non-strict JavaScript and a special object inside non-arrow functions. Renaming it via destructuring is the safest pattern; the body field name stays `arguments` (which is what the SDK expects).
- The Zod schema field `model: z.string().optional()` — single string, NOT `z.object({ providerID, modelID })`. This is the deliberate API difference per D-19. Read the input schema description literally; do NOT "harmonize" with opencode_run.
- The SDK method is `client.session.command` (verified in node_modules/@opencode-ai/sdk/dist/gen/sdk.gen.d.ts line 391).
- Conditional spread for all three optional fields (matches the project's existing pattern).
- Return value is `JSON.stringify(data)` — same as Phase 3 tools. Do NOT validate parts against PartSchema here; the SURF-02 work for this tool is deferred until a downstream caller needs it (none in this phase).

Do NOT modify any other tool, the imports, the constants, or the main() block.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| diff library → MCP server | The `diff` package runs locally on `before`/`after` strings the OpenCode server returned; pure JS, no shell out, no native code. |
| Claude Code → opencode_session_command | command and arguments strings cross the trust boundary; the OpenCode server is the actual interpreter and validator of slash commands. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-08 | Tampering | createPatch input strings (before, after) | accept | Inputs come from the local OpenCode server (trust domain). The `diff` library is pure JS with no native bindings (per D-04) and no shell-out behavior. Worst-case malformed input produces a malformed patch string, not RCE. |
| T-04-09 | Denial of Service | createPatch on very large files | accept | OpenCode itself caps file sizes; `diff` is O(N*M) Myers but bounded by what OpenCode returns. For a local dev tool this is acceptable; if a giant file is opened we'd see latency on the get_diff call, not a security issue. |
| T-04-10 | Elevation of Privilege | opencode_session_command runs slash commands | mitigate | The tool only forwards `command` + `arguments` to OpenCode. Slash commands are interpreted by OpenCode itself, not by Prefect. Per CLAUDE.md, OpenCode permissions are auto-approved (the established trust model — git is the safety net). No privilege escalation introduced beyond what the user already grants OpenCode. |
| T-04-11 | Tampering | model field plain-string vs object confusion | mitigate | Zod schema for `opencode_session_command.model` is `z.string().optional()` while `opencode_run.model` is `z.object({...})`. Schema enforces the distinction at MCP boundary so callers cannot send the wrong shape to the wrong endpoint. |
| T-04-12 | Information Disclosure | patch field exposes full file before/after content | accept | The MCP caller is the same developer who has read access to the file. No new exposure beyond what `opencode_get_diff` already returned. |
</threat_model>
