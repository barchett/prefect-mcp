```typescript
  'opencode_list_agents',

  async ({ directory }) => {

      return { content: [{ type: 'text', text: String(err) }], isError: true };
    }
  }
);
```

```typescript
// API-03: Find workspace symbols by query (Phase 8)
server.registerTool(
  'opencode_find_symbol',
  {
    description: 'Search the OpenCode workspace for symbols matching a query string (e.g. function or class names). Returns Array<{ name, kind, path, range }> where path is project-root-relative when a directory is resolved (via directory param or OPENCODE_DEFAULT_PROJECT), absolute otherwise. kind is the LSP SymbolKind number.',
    inputSchema: z.object({
      query: z.string().describe('Symbol name or pattern to search for'),
      directory: z.string().optional().describe('Absolute path to the project root. Falls back to OPENCODE_DEFAULT_PROJECT env var if not provided.'),
    }),
  },
  async (args) => {
    const { query: symbolQuery, directory } = args;
    const dir = resolveDirectory(directory);
    try {
      const { data, error } = await client.find.symbols({
        query: { query: symbolQuery, ...(dir ? { directory: dir } : {}) },
      });
      if (error) throw new Error(JSON.stringify(error));
      const mapped = (data ?? []).map((sym) => {
        const absolutePath = sym.location.uri.replace(/^file:\/\//, '');
        const filePath = dir ? path.relative(dir, absolutePath) : absolutePath;
        return {
          name: sym.name,
          kind: sym.kind,
          path: filePath,
          range: sym.location.range,
        };
      });
      return { content: [{ type: 'text', text: JSON.stringify(mapped) }] };
    } catch (err) {
      return { content: [{ type: 'text', text: String(err) }], isError: true };
    }
  }
);
```

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| MCP client (Claude Code) -> Prefect MCP server | Tool input args (`directory`, `query`) come from caller — already trusted in single-user personal-tool model, but still validated by Zod |
| Prefect MCP server -> OpenCode HTTP API | Outbound calls via `createOpencodeClient` + `fetchWithAuth` — auth header injected upstream by Phase 6's `fetchWithAuth` wrapper |
| OpenCode response -> Prefect response transform | Untrusted-shape JSON crossing back into our process; we trust the SDK's typed parse but apply field projection before returning |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-08-01 | Information Disclosure | `opencode_list_providers` response | mitigate | D-04 trim: surface only `{ id, name }` for each model — strip `release_date`, cost, limits, capabilities, npm package metadata. Limits surface area of model registry exposed to MCP clients to what's actually needed for `providerID`/`modelID` selection. |
| T-08-02 | Information Disclosure | `opencode_find_symbol` returned `path` | mitigate | D-06 path relativization: when a project root is resolved, strip `file://` and convert to project-relative path. Avoids leaking absolute filesystem layout (e.g., `/home/<username>/...`) to MCP clients when callers have already specified a project context. D-07: when no project root is known, fall back to absolute path (do not manufacture relative path from `process.cwd()` — Phase 5 locked decision). |
| T-08-03 | Information Disclosure | `opencode_list_agents` response | mitigate | D-01 field projection: surface only `{ name, description, mode }` — strip `builtIn`, `permission` config, internal model defaults. Same reduction-of-exposure rationale as T-08-01. |
| T-08-04 | Tampering / Injection | `opencode_find_symbol` `query` Zod input forwarded to OpenCode `/find/symbol` endpoint | accept | Zod enforces `z.string()`. The string is passed as a `query.query` SDK arg — the SDK serializes it as a URL query parameter, not interpolated into a shell or SQL context. OpenCode (localhost service) is responsible for its own search engine input validation. No injection vector visible at our boundary; documenting acceptance per ASVS L1. |
| T-08-05 | Spoofing / Auth | Auth propagation to OpenCode for the three new endpoints | mitigate | All three tools use the existing module-scope `client` instance (line 15 of src/index.ts), which was created with `fetch: fetchWithAuth` in Phase 6 (CORE-08 + INFRA-04/05). | T-08-06 | Denial of Service | Large symbol-search results from `opencode_find_symbol` consuming MCP response budget | accept | Personal-tool model; OpenCode's `/find/symbol` already returns a bounded result set scoped to workspace indexing. No pagination requested for v3.0 (CONTEXT.md scope: read-only wrappers, no extra ergonomics). If sizes become a problem, future iteration can add a `limit` param — out of scope here. |
| T-08-07 | Repudiation | Audit trail for these read-only calls | accept | Read-only operations, no state change in OpenCode or local filesystem. No audit logging required for ASVS L1 read-only personal-tool baseline. |
