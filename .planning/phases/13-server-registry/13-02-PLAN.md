## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| process.argv → handleAddServer | User-supplied CLI arguments cross into the registry write path; arbitrary strings for name/host/model, parsed integer for port |
| Spawned cli.js subprocess in tests → real ~/.config/prefect/ | Without HOME redirect, integration tests would corrupt the user's actual registry |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-13-06 | Tampering | port arg parsed into URL components downstream | mitigate | handleAddServer validates port with `parseInt(portStr, 10)` and `Number.isFinite(port) && port >= 1 && port <= 65535`; rejects with stderr error and exit 1 otherwise. This prevents `NaN` or out-of-range integers reaching Phase 14's URL builder. |
| T-13-07 | Information Disclosure | Test suite writing to real ~/.config/prefect/ | mitigate | Every integration test passes `env: { HOME: dir, USERPROFILE: dir }` to spawnSync, redirecting `homedir()` inside the child process to the freshTmp dir. Tests cleanup via `rmSync(dir, { recursive: true, force: true })` in `finally`. |
| T-13-08 | Denial of Service | Bogus or oversized argv input | accept | Node's argv parsing is bounded by OS argv limits; not a Prefect threat surface. The CLI immediately exits on any unrecognized subcommand or malformed args. |
| T-13-09 | Spoofing | name field collision (e.g., adding 'local' twice) | mitigate | registry.addServer (plan 13-01) detects name collision and overwrites with stderr warning. The user sees `Updated existing server '<name>'.` This is RESEARCH A1 confirmed behavior — not a silent data loss. |
| T-13-10 | Tampering | Shell metacharacters in name/host/model arguments | accept | These strings are written into JSON (not shell) and read back into the Phase 14 fetch() URL builder, which percent-encodes or rejects malformed components. No shell expansion path exists in either Plan 13-01 or 13-02 code. |
</threat_model>
