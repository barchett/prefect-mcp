## Fixed Issues (iteration 1)

### WR-02: Missing break statements in switch block — fall-through from add-server into remove-server

**Status:** Fixed
**Commit:** 037ec44
**How:** Added `break` after `case 'add-server'`, `case 'remove-server'`, and `case 'list-servers'` in `src/cli.ts`. TypeScript does not warn about this because each handler has return type `never` (calls `process.exit`), but the breaks prevent fall-through if handlers throw before reaching `process.exit`.

### WR-03: parseInt accepts floats for --max-sessions flag

**Status:** Fixed
**Commit:** 037ec44
**How:** Replaced `parseInt`-based validation with a `/^\d+$/` regex check against the raw string, followed by a `< 1` check on the parsed value. This rejects `'1.9'`, `'1.5'`, and other decimal inputs that `parseInt` would silently truncate.

## Fixed Issues (iteration 2)

### WR-01: Race condition in capacity enforcement — concurrent session creation overshoots maxSessions

**Status:** Fixed
**Commit:** 5f37353
**How:** Added `proper-lockfile` dependency. In `sessions.ts`: added `withSessionLock<T>` (acquires a 30 s stale-safe file lock on `sessions.json`) and exported `atomicCheckAndAdd` which runs the full read → count → check → write critical section under that lock. In `handlers.ts`: `createSession` accepts a new `maxSessions?: number | null` param; when set, it calls `atomicCheckAndAdd` after the OpenCode HTTP call — if the race is lost (capacity exceeded by a concurrent instance), it aborts the just-created session via `client.session.delete` and throws the capacity error. When `maxSessions` is undefined, the existing sync `addSession` path is used unchanged. In `index.ts`: all three `createSession` call sites (`prefect_create_session`, `prefect_delegate`, `prefect_dispatch`) now thread `serverEntry?.maxSessions` through. The existing optimistic `checkCapacity` pre-check is retained as a fast-path to avoid an unnecessary HTTP call when the server is obviously at capacity.

### IN-01: removeServer writes to console.error instead of console.log for success confirmation

**Status:** Fixed
**Commit:** 5f37353
**How:** Changed `console.error(`Removed server '${name}'.`)` to `console.log` in `src/registry.ts`. Updated the corresponding `cli.test.ts` assertion from `stderr` to `stdout`.

### IN-02: listServers uses console.log while diagnostic output uses console.error

**Status:** Fixed
**Commit:** 5f37353
**How:** Changed `console.error(`Updated existing server '${entry.name}'.`)` in `addServer` (`src/registry.ts`) to `console.log`, making success confirmations consistent — data and confirmations go to stdout, errors and warnings stay on stderr.
