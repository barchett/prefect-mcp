# Phase 15.1 Plan 02: Capacity Enforcement in src/index.ts Summary

**One-liner:** checkCapacity() helper guards prefect_create_session, prefect_delegate (new-session path), and prefect_dispatch (new-session path) ŌĆö returning "at capacity" errors when active sessions >= maxSessions, leaving sessionId-reuse paths unrestricted.

## What Was Built

### Task 1: checkCapacity helper and three call-site guards in src/index.ts

- Updated `countSessionsForServer` import from `./sessions.js` (line 13)
- Added `checkCapacity(serverName, serverEntry)` helper function after `serverNameForUrl` (line 92-102): returns an error string with `"at capacity"`, current count, and max when `active >= serverEntry.maxSessions`; returns `undefined` when `serverEntry` is absent or `maxSessions` is `null`/`undefined` (unlimited)
- Guarded `prefect_create_session`: inserted capacity check after `model` resolution and before `createSession()` call; returns `isError: true` if at capacity
- Guarded `prefect_delegate` create-new-session path: inserted capacity check after `getClient()` and before `createSession()`; calls `clearTimeout(timer)` before returning error to prevent dangling timer
- Guarded `prefect_dispatch` create-new-session path: inserted capacity check after `getClient()` and before `createSession()`
- The `providedSessionId` reuse branches in both `prefect_delegate` and `prefect_dispatch` have NO capacity check ŌĆö they operate on existing sessions without creating new ones
