# Phase 15.1: Code Review Report

**Reviewed:** 2026-05-03T00:00:00Z
**Depth:** standard
**Files Reviewed:** 6
**Status:** issues_found

## Summary

This phase implements MULTI-11: a `maxSessions` field on server registry entries and capacity enforcement in `prefect_create_session` and `prefect_delegate`/`prefect_dispatch`. The data model, persistence, CLI flag, and `listServers` column are all correct and well-tested. The capacity check helper (`checkCapacity`) is correctly placed and its logic is sound.

Three warnings are present:

1. A race condition in the capacity check — count and create are non-atomic, so concurrent `prefect_create_session` calls can both pass the guard and overshoot `maxSessions`.
2. Missing `break` statements in a `switch` block in `cli.ts`, which causes unintentional fall-through from `add-server` into `remove-server` and from `remove-server` into `list-servers`.
3. The `--max-sessions` flag parser is silently vulnerable to float inputs (e.g. `--max-sessions 1.9` passes as `1`).

Two info items: `removeServer` writes to `console.error` instead of `console.log` for the success confirmation, and `listServers` uses `console.log` while all other diagnostic output in the codebase uses `console.error` (stdout vs stderr inconsistency).

## Warnings

### WR-01: Race condition in capacity enforcement — concurrent session creation overshoots `maxSessions`

**File:** `src/index.ts:92-101` (also `src/index.ts:128-133`, `src/index.ts:993-998`, `src/index.ts:1082-1084`)

**Issue:** `checkCapacity` reads `countSessionsForServer` from `sessions.json`, then (if under limit) proceeds to `createSession` which writes a new entry back to `sessions.json`. These are two separate filesystem reads — there is no lock. If two concurrent `prefect_create_session` calls both read the count before either has written the new session entry, both pass the `active >= maxSessions` guard and both create a session, resulting in `maxSessions + 1` active sessions. For a single-process MCP server this is low-probability but not impossible under parallel tool calls from an orchestrator.

**Fix:** Move the session registration (`addSession`) to happen inside the same synchronous read-modify-write cycle that `countSessionsForServer` uses, or perform an optimistic count increment before the async `createSession` call and decrement on failure:

```typescript
// Option A: count + reserve atomically in sessions.json before the async call
function reserveSession(serverName: string, sessionId: string, entry: SessionEntry, sessionsPath?: string): 'ok' | 'at_capacity' {
  const map = readSessionMap(sessionsPath);
  const active = Object.values(map.sessions).filter((e) => e.server === serverName).length;
  const reg = readRegistry();
  const serverEntry = reg.servers.find((s) => s.name === serverName);
  if (serverEntry?.maxSessions != null && active >= serverEntry.maxSessions) {
    return 'at_capacity';
  }
  map.sessions[sessionId] = entry;
  writeSessionMap(map, sessionsPath);
  return 'ok';
}
```

This collapses the check + write into one atomic (sync) file operation so a second concurrent call sees the incremented count.

### WR-02: Missing `break` statements in `switch` block — fall-through from `add-server` into `remove-server` and `list-servers`

**File:** `src/cli.ts:201-208`

**Issue:** The `switch (subcommand)` statement has no `break` after `case 'add-server'` or `case 'remove-server'`. TypeScript does not warn about fall-through in `switch` by default. Each handler calls `process.exit(0)` on the success path, so the fall-through is masked at runtime. However, if a handler throws before reaching `process.exit` (e.g., an unexpected exception in `handleAddServer`) control falls through to `handleRemoveServer`, which will then run with the wrong arguments and may produce a misleading error. This is a latent correctness bug and violates expected switch semantics.

```typescript
// Current (lines 201-208):
case 'add-server':
  handleAddServer(args.slice(1));   // no break — falls through to remove-server
case 'remove-server':
  handleRemoveServer(args.slice(1)); // no break — falls through to list-servers
case 'list-servers':
  handleListServers();
default:
  usageAndExit();
```

**Fix:** Add `break` after each case (or explicitly annotate intentional fall-through):

```typescript
case 'add-server':
  handleAddServer(args.slice(1));
  break;
case 'remove-server':
  handleRemoveServer(args.slice(1));
  break;
case 'list-servers':
  handleListServers();
  break;
default:
  usageAndExit();
```

Note: `handleAddServer`, `handleRemoveServer`, `handleListServers`, and `usageAndExit` are all typed `never` (they each call `process.exit`), so TypeScript does not flag the missing breaks as unreachable. A linter rule (`no-fallthrough`) would catch this.

### WR-03: `parseInt` accepts floats silently — `--max-sessions 1.9` is accepted as `1`

**File:** `src/cli.ts:87-90`

**Issue:** `parseInt('1.9', 10)` returns `1`, which passes the `Number.isFinite(parsed) || parsed < 1` guard and stores `1` in the registry. A user who mistakenly types `--max-sessions 1.9` gets no error and an unexpected value. The same issue exists for inputs like `1e2` (`parseInt('1e2', 10)` = `1`, not `100`).

```typescript
// Current (line 87-90):
const parsed = parseInt(maxSessionsStr ?? '', 10);
if (!Number.isFinite(parsed) || parsed < 1) {
```

**Fix:** Validate that the raw string matches a positive integer before parsing:

```typescript
const maxSessionsStr = handlerArgs[maxSessionsIdx + 1] ?? '';
if (!/^\d+$/.test(maxSessionsStr) || maxSessionsStr === '0') {
  console.error(`Error: invalid --max-sessions '${maxSessionsStr}' — must be a positive integer`);
  process.exit(1);
}
const maxSessions = parseInt(maxSessionsStr, 10);
if (maxSessions < 1) {
  console.error(`Error: invalid --max-sessions '${maxSessionsStr}' — must be a positive integer`);
  process.exit(1);
}
```

## Info

### IN-01: `removeServer` logs success message to `console.error` instead of `console.log`

**File:** `src/registry.ts:60`

**Issue:** `console.error(`Removed server '${name}'.`)` sends the success confirmation to stderr. This is inconsistent with `listServers` which uses `console.log` for its output, and it forces callers to capture stderr for the success message. The convention in this codebase appears to be: diagnostic/warning output to `console.error`, user-facing results to `console.log`. A success confirmation for a mutation (`remove-server`) is borderline, but `addServer` on line 44 similarly uses `console.error` for "Updated existing server". At minimum the two mutation helpers should be consistent with each other; consider whether success output should go to stdout.

**Fix:** Decide on a convention and apply it consistently. If mutation confirmations should go to stderr (so scripts can redirect stdout cleanly), document it. If they should go to stdout, change both `addServer` (line 44) and `removeServer` (line 60) to `console.log`.

### IN-02: `listServers` column header check in `registry.test.ts` does not assert `CAPACITY` column content for existing entries

**File:** `src/registry.test.ts:124-142`

**Issue:** The existing `listServers prints tabular header + rows to stdout when entries exist` test (lines 124-142) was written before MULTI-11. It asserts on `NAME`, `PORT`, `PROVIDER`, `MODEL` but does NOT assert on `CAPACITY`. The new MULTI-11-specific tests (`MULTI-11: listServers shows CAPACITY column header and unlimited for uncapped server`, lines 217-228) cover this separately using the build artifact driver, which is correct. However, the older integration-style test at line 124 leaves a gap: it would pass even if the `CAPACITY` header were accidentally removed from non-MULTI-11 codepaths. This is a minor test coverage gap, not a defect in production code.

**Fix:** Add an assertion to the existing test at line 133 (or nearby):

```typescript
assert.ok(res.stdout.includes('CAPACITY'), `stdout missing CAPACITY, was: ${res.stdout}`);
```
