# Milestone v2.0: Session Management + Run Options + Infrastructure

**Status:** ✅ SHIPPED 2026-04-27
**Phases:** 3–4
**Total Plans:** 6

## Overview

Expand Prefect's toolset from 7 tools to 18 — adding full session lifecycle management (9 new tools), richer `opencode_run` options (model/agent/system overrides), a fire-and-forget `opencode_prompt_async`, a more reliable AbortController timeout, structured response surfaces (PartSchema, patch field), and a zero-friction `prefect init` CLI.

## Phases

### Phase 3: Session Management Tools

**Goal:** Claude Code can inspect, navigate, and manage OpenCode sessions without leaving the MCP workflow.
**Depends on:** Phase 2 (v1.0 MCP server baseline)
**Requirements:** SESSION-01, SESSION-02, SESSION-03, SESSION-04, SESSION-05, SESSION-06, SESSION-07, SESSION-08, SESSION-09
**Plans:** 2 plans

Plans:
- [x] 03-01: Read-only session tools — `opencode_session_list`, `opencode_session_get`, `opencode_session_status`, `opencode_session_messages`, `opencode_session_message` (SESSION-01 to SESSION-05)
- [x] 03-02: Mutating session tools — `opencode_session_delete`, `opencode_session_rename`, `opencode_session_children`, `opencode_session_unrevert` (SESSION-06 to SESSION-09)

**Success Criteria (all met):**
1. Claude Code can list all sessions and identify one by ID, title, or directory without manual API calls ✓
2. Claude Code can retrieve full message history for a session (all messages or limited slice), and fetch a single message by ID ✓
3. Claude Code can check real-time session status (idle/busy/retrying) across all active sessions before deciding to call `opencode_run` ✓
4. Claude Code can delete a session it no longer needs and rename a session for clarity ✓
5. Claude Code can list child sessions of a forked session and unrevert a session to undo a prior revert ✓

### Phase 4: Run Options + Structured Responses + Infrastructure

**Goal:** `opencode_run` is the reliable, feature-complete backbone of the Prefect workflow — supporting model/agent/system overrides, async fire-and-forget, structured response surfaces, and a correct timeout that actually cancels in-flight requests.
**Depends on:** Phase 3
**Requirements:** RUN-01, RUN-02, RUN-03, RUN-04, SURF-01, SURF-02, INFRA-01, INFRA-02, CMD-01
**Plans:** 4 plans

Plans:
- [x] 04-01: Zod schemas for the 12-member Part discriminated union (SURF-02 schemas in `src/parts.ts`)
- [x] 04-02: `opencode_run` body fields (RUN-01/02/03), AbortController timeout (INFRA-01), `prompt_async` tool (RUN-04), parts validation (SURF-02)
- [x] 04-03: `opencode_get_diff` patch field via `diff` package (SURF-01), `opencode_session_command` tool (CMD-01)
- [x] 04-04: `prefect init` CLI with merge-not-overwrite `.mcp.json` (INFRA-02)

**Success Criteria (all met):**
1. Claude Code can run a prompt against a specific non-default model by passing `providerID` + `modelID` together ✓
2. Claude Code can inject a custom system prompt for a single prompt without affecting session configuration ✓
3. Claude Code can fire a prompt and return immediately without blocking, using `opencode_prompt_async` ✓
4. `opencode_get_diff` returns a top-level `patch` string field; `opencode_run` returns a structured `parts` array ✓
5. A timed-out `opencode_run` cancels the in-flight HTTP connection (AbortController, not Promise.race) ✓
6. A developer can run `prefect init` to write a correct `.mcp.json` without manual JSON editing ✓
7. Claude Code can execute slash commands inside a session via `opencode_session_command` ✓

## Milestone Summary

**Key Decisions:**
- `SESSION-03` (`opencode_session_status`) has no sessionId param — it is a global endpoint returning all sessions
- `SESSION-04` limit uses explicit `!== undefined` check (not falsy) to correctly handle edge cases
- `SESSION-05` `messageId` (MCP) maps to `messageID` (SDK path param) — case difference is intentional per SDK types
- SURF-02 discriminator strings verified from `node_modules/@opencode-ai/sdk/dist/gen/types.gen.d.ts` — all 12 Part type literals encoded before writing schemas
- INFRA-01 (AbortController) and RUN-04 (prompt_async) implemented atomically — both touch the async/timeout path in `opencode_run`
- `opencode_prompt_async` uses `client.session.promptAsync` (separate SDK method), returns `{ sessionId, accepted: true }`, no AbortController
- CLI entry point: separate `src/cli.ts` compiles to `build/cli.js` — avoids coupling MCP server startup with CLI argument parsing
- Manual `process.argv` parsing over Commander.js — minimal surface (one subcommand, one flag), zero additional deps
- `model: z.string().optional()` for `opencode_session_command` (not z.object) — session.command endpoint takes a single string
- `arguments` destructured to `args` in `opencode_session_command` — `arguments` is reserved in non-strict JS; body field stays `'arguments'` per SDK
- install script (`curl | bash`) deferred entirely to v3.0 as `npm install -g` (requires npm publish)

**Issues Resolved:**
- Promise.race timeout orphaned in-flight TCP connections — replaced by AbortController on fetch signal
- `data ?? []` guard needed in `opencode_get_diff` — SDK types `data` as possibly undefined
- `prefect init` absolute path resolution required `fileURLToPath(import.meta.url)` for ESM compatibility

**Issues Deferred:**
- install script (`npm install -g prefect-mcp`) — requires npm publish, deferred to v3.0
- `opencode_run` `tools` override, `FilePartInput`, `messageID`, `AgentPartInput`/`SubtaskPartInput` — deferred to Phase 5 (v3.0)

**Technical Debt:**
- REQUIREMENTS.md SESSION-01 through SESSION-09 checkboxes not updated during Phase 3 execution (PROJECT.md was source of truth) — documentation gap only, all requirements implemented and validated
